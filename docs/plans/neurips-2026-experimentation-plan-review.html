재현 가능한 RAG 재구현 프레임워크 실험 계획

이 문서는 NeurIPS 2026 / TMLR / JMLR 제출을 염두에 둔 실험 계획을 검토하기 쉽게 재구성한 버전입니다. 핵심은 “평가 지표 고발”이 아니라, AutoRAG-Research가 다른 RAG 연구를 재구현하고 재현 가능한 실험 artifact를 만드는 프레임워크임을 실험으로 입증하는 것입니다.

1. 논문 방향성

논문의 주장을 어디에 둘지 가장 먼저 고정해야 합니다.

중심 논문 주장 AutoRAG-Research는 RAG 연구의 재현성을 높이기 위해, published retrieval/generation pipeline을 공통 abstraction으로 재구현하고, benchmark dataset을 공통 schema로 ingest하며, config 기반으로 실험을 실행하고, 결과 artifact를 배포할 수 있게 하는 재구현 프레임워크이다.

강조해야 할 것

Reimplementation fidelity Artifact reproducibility Common schema Config-driven execution Extensibility
  • 어떤 method가 어떤 shared abstraction으로 표현되는가
  • 어떤 실험 artifact로 figure/table을 재생성할 수 있는가
  • 어떤 deviation이 문서화되었는가

메인 스토리로 피해야 할 것

Metric critique 중심 Ranking instability 중심 SOTA 주장 완전한 역사적 score 복제
  • “RAG metric이 틀렸다”를 논문 제목/핵심 주장으로 삼지 않기
  • 단일 benchmark ranking으로 pipeline 우열을 과도하게 일반화하지 않기
  • closed model / prompt / preprocessing 차이로 인해 exact numeric reproduction은 non-claim으로 처리

2. 연구 질문

각 RQ는 실험 결과 표와 artifact evidence로 바로 연결되어야 합니다.

1
RQ1 — Framework Reimplementation Coverage 한 framework가 다양한 published RAG retrieval/generation pipeline을 method-specific harness 없이 표현할 수 있는가?
2
RQ2 — Reproducible Execution and Artifact Reuse config, DB dump, embedding, cached output, result artifact만으로 table/figure을 재현할 수 있는가?
3
RQ3 — Empirical Benchmarking Under a Shared Protocol 대표 RAG pipeline을 같은 protocol에서 돌렸을 때 어떤 성능, 비용, runtime, failure pattern이 나타나는가?
4
RQ4 — Extensibility for Future Reimplementation 새로운 method, metric, dataset을 추가할 때 얼마나 적은 harness 변경으로 executor/evaluator/reporting 흐름에 참여하는가?

4. Dataset Subset / Corpus 정책

큰 dataset은 deterministic subset으로 평가하되, 작은 dataset은 그대로 사용합니다.

핵심 정책 Query가 이미 2,000–3,000개 이하인 test split은 full로 사용합니다. 그보다 큰 dataset은 retrieval-only에서는 query_limit=3000, generation/VLM처럼 비용이 큰 실험에서는 query_limit=2000을 기본값으로 둡니다.

Query subset

  • 대형 retrieval-only benchmark: query_limit=3000
  • generation / judge / multimodal: query_limit=2000
  • 작은 dataset은 full split 유지
  • seed 42와 selected query IDs 공개

Corpus target

  • 현재 ingestor API의 corpus control은 min_corpus_cnt
  • corpus_limit이라는 hard cap으로 표현하지 않기
  • gold/required corpus를 먼저 포함
  • 그 뒤 non-gold corpus를 seed 기반 sampling
min_corpus_cnt 주의점 이것은 strict maximum이 아닙니다. 선택된 query의 gold/required corpus 수가 target보다 많으면 최종 corpus 수는 min_corpus_cnt보다 커질 수 있습니다. 논문과 manifest에서는 “corpus target” 또는 “corpus subset target”으로 설명하는 것이 안전합니다.
Dataset family Query subset 정책 Corpus subset 정책
BEIR / MTEB 작은 task는 full, 큰 task는 query_limit=3000 min_corpus_cnt 지원. qrel/gold document 항상 보존
MrTyDi English test query는 full 가능. 더 큰 split/language aggregate는 cap 검토 full corpus indexing 부담 시 min_corpus_cnt 사용
BRIGHT single domain은 full. all-domain aggregate는 비용에 따라 domain-stratified 가능 short-document corpus가 클 때 min_corpus_cnt 사용. long-doc mode는 상대적으로 작음
RAGBench techqa/covidqa는 full. 대형 config/all-config는 query_limit=2000 min_corpus_cnt ineffective. documents가 example 단위로 bundle됨
CRAG main run은 query_limit=2000 또는 3000 min_corpus_cnt ineffective. search context가 query-specific
ViDoRe v1 subsampled dataset은 full 1:1 query-image 구조라 query_limit과 min_corpus_cnt가 사실상 동일
ViDoRe v2/v3 / VisRAG single config는 대부분 full. all-config aggregate는 stratified/capped shared corpus가 있는 ingestor에서는 min_corpus_cnt 지원
Open-RAGBench selected path가 cap을 넘으면 query_limit 사용 min_corpus_cnt ineffective. query-to-section 구조

5. Pipeline 선택

Full-scale 실험에서는 구현된 pipeline을 제외하지 않습니다. 비용은 subset, cache, staged execution으로 제어하고, 결과 표가 너무 크면 appendix/detail artifact로 보냅니다.

Retrieval pipelines — full-scale 포함

All implemented where valid
  • BM25 — sparse lexical baseline, deterministic sanity check
  • Vector Search — dense retrieval baseline
  • Hybrid RRF — compositional retrieval wrapper
  • Hybrid CC — score-level sparse+dense fusion
  • HyDE — LLM-assisted retriever reimplementation
  • Query Rewrite — reusable LLM-backed preprocessing
  • RETRO* — high-cost LLM reranking case, full-scale required
  • Power of Noise — retrieval perturbation/diagnostic case
  • Question Decomposition Retrieval — sub-query retrieval case
  • HEAVEN — visual retrieval on multimodal datasets

주의 RETRO*는 비싸지만 제외 대상이 아니라 cache/token manifest가 필요한 high-cost case입니다.

Generation pipelines — full-scale 포함

All implemented where valid
  • BasicRAG — retrieve-then-generate baseline
  • IRCoT — interleaved retrieval + reasoning
  • ET2RAG — ensemble/subset selection
  • MAIN-RAG — multi-agent filtering
  • Question Decomposition — query decomposition abstraction
  • Self-RAG — self-reflective generation
  • RAG-Critic — critic-guided correction/revision
  • SPD-RAG — speculative/parallel RAG strategy
  • AutoThinkRAG — adaptive reasoning, including multimodal variants where valid
  • VisRAG Generation — vision-language generation on visual/multimodal datasets

상세 결과가 너무 크면 appendix에 배치하되, full-scale artifact에는 모든 modality-compatible run을 포함합니다.

6. 재현성 프로토콜

이 논문의 핵심 평가 대상은 model score뿐 아니라 artifact 자체입니다.

필수 manifest

Manifest 필수 기록 항목 왜 중요한가
Config manifest experiment YAML, pipeline YAML, metric YAML 정확한 실험 재구성
Dataset manifest ingestor, split/subset, query_limit, min_corpus_cnt, selected query/corpus IDs, embedding model, dump hash preprocessing 재현성
Environment manifest git commit, lockfile, Python, Docker, PostgreSQL, VectorChord version 실행 환경 audit
LLM manifest provider, model name/version, temperature, max tokens, prompt template, cache key closed/remote model drift 대응
Result manifest retrieval rows, generation rows, metric rows, artifact hashes figure/table 재생성

Reproduction modes

Mode Required inputs Target time Expected output
Figure-only result parquet/csv + figure scripts < 10분 논문 figure/table 재생성
Small public rerun one DB dump + configs + cached/cheap model path < 1시간 subset score 재계산
Full rerun all datasets + embeddings + API/local models days/weeks full benchmark regeneration

7. 분석 계획

benchmark score는 framework validation evidence입니다. 논문의 주객이 뒤집히지 않도록 구성합니다.

A
Framework coverage and fidelity pipeline inventory, dataset inventory, metric inventory, fidelity/deviation cards를 표로 제시.
B
Reproducibility evaluation figure-only reproduction time, small rerun time, deterministic retrieval agreement, metric recomputation agreement, artifact completeness를 측정.
C
Empirical benchmark results retrieval leaderboard, generation leaderboard, runtime/cost/token usage, per-query failure examples을 보고.
D
Extensibility case study 작은 plugin/pipeline/metric 추가 사례를 통해 files changed, LOC, tests, config effort를 보고.
Metric sensitivity의 위치 metric에 따라 ranking이 바뀌는 현상은 appendix 또는 secondary analysis로 두는 것이 안전합니다. 이 논문의 primary evidence는 “재현 가능한 재구현 framework와 artifact package”입니다.

Risk Impact Mitigation

exact original score reproduction이 안 됨 High exact score matching은 non-claim. deviation/fidelity card로 투명하게 처리
generation metric을 answer 없는 dataset에 사용 High retrieval-only vs generation track 분리. generation_gt 필요 조건 명시
pipeline/config contamination Medium unique pipeline names + config manifest + DB result hash 관리
full benchmark 비용 과다 Medium 모든 pipeline은 포함하되 deterministic capped runs, cached outputs, staged execution, token/runtime manifest로 비용 관리
closed model drift Medium cached outputs, model version, temperature 0, prompt/cache manifest 공개

9. 검토 체크리스트

이 HTML을 보면서 최종 실험 계획에서 빠진 결정을 체크하기 위한 항목입니다.

  • 논문 제목과 abstract가 “metric critique”로 읽히지 않는가?
  • RQ1–RQ4가 각각 table/figure로 증명 가능한가?
  • BEIR/MrTyDi를 generation metric에 사용하지 않도록 명확히 막았는가?
  • CRAG에 query_limit=2000/3000 정책을 적용했는가?
  • MrTyDi corpus에 min_corpus_cnt 필요 여부를 결정했는가?
  • corpus_limit이라는 hard cap 표현을 피했는가?
  • 각 subset의 query IDs/corpus IDs/seed를 manifest로 남길 수 있는가?
  • 각 pipeline의 fidelity/deviation card 초안이 있는가?
  • figure-only reproduction path가 실제로 10분 내 가능한가?
  • small public rerun dataset과 config가 하나로 좁혀졌는가?
  • LLM output cache와 prompt/model manifest가 준비되는가?
  • full-scale artifact에 모든 구현 pipeline의 modality-compatible run이 포함되는가?
